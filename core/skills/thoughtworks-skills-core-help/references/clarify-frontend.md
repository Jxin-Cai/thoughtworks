# 前端需求澄清场景指令

## 项目上下文扫描

在向用户提任何问题之前，先扫描项目现状，目标是了解前端技术栈、已有页面结构和后端 API 契约，避免问出已有答案的问题：

1. **扫描前端项目结构** — 使用 Glob 工具查看前端目录（`src/app/`、`src/pages/`、`src/features/`、`src/entities/`、`src/shared/`、`src/widgets/`、`package.json` 等），目标是确定前端技术栈和已有页面布局
2. **读取关键文档** — 使用 Read 工具读取项目的 `README.md`、`CLAUDE.md`（如存在），目标是获取前端架构约定和开发规范
3. **查看最近提交** — 使用 Bash 执行 `git log --oneline -10`，目标是了解最近的前端开发动态
4. **读取 OHS 导出契约** — 读取 `.thoughtworks/<idea-name>/backend-designs/ohs.md`，目标是了解后端提供的 API 端点，作为前端需求澄清的基础
5. **扫描已有前端代码** — 使用 Glob 查找 `**/src/pages/**/*.tsx`、`**/src/features/**/*.{ts,tsx}`、`**/src/entities/**/*.{ts,tsx}`、`**/src/shared/api/**/*.ts`，目标是了解已有的页面和 API 调用模式
6. **判断是否为 0-1 项目** — 检查 `src/` 目录是否存在、`package.json` 是否含前端依赖（如 `react`、`vue` 等）。如果 `src/` 不存在或无前端依赖，标记为 **0-1 新项目**
7. **判断已有项目的架构模式** — 如果非 0-1 项目，检查目录结构：
   - 存在 `src/features/` 或 `src/entities/` → 已使用 **FSD 架构**
   - 存在 `src/pages/` + `src/components/` + `src/api/`（无 features/entities）→ 使用**平铺目录结构**
   - 记录当前架构模式供后续提问使用

将扫描结果整理为内部参考（不输出给用户），用于指导后续提问。

## 澄清维度清单

以下维度是**引导锚点**，每个维度都必须深挖到足以指导前端设计的精度，而不是得到一个笼统回答就标记通过。

### 1. 页面需求

入口问题：需要哪些页面？每个页面的主要功能？

深挖方向：
- 用户说"一个管理页面" → 追问"管理什么？列表展示哪些字段？支持搜索和筛选吗？有分页吗？"
- 用户只提了主要页面 → 追问"除了列表页，还需要详情页吗？创建/编辑是弹窗还是独立页面？"
- 结合 OHS 契约追问"后端提供了 X 和 Y 两个接口，对应的页面结构你的预期是什么？"

### 2. 交互需求

入口问题：有哪些关键交互流程？

深挖方向：
- 用户说"提交表单" → 追问"提交前需要校验哪些字段？提交后跳转到哪里？需要显示加载状态吗？失败后如何提示？"
- 用户提了列表 → 追问"列表支持哪些操作？批量操作？行内编辑？排序？"
- 构造具体场景验证："用户填到一半刷新了页面，数据需要保留吗？"

### 3. 技术栈

入口问题：使用什么前端框架？

深挖方向：
- 如项目已有前端代码，确认沿用："项目当前使用 React + TypeScript，新功能是否沿用？"
- 用户说"用 React" → 确认"React + TypeScript？路由用什么方案？"
- React + TypeScript 为默认，如项目已有前端代码则沿用已有技术栈

### 4. UI 约束与风格

入口问题：有没有设计稿？使用什么 UI 组件库？有没有特殊的样式要求？

深挖方向：
- 用户说"没有设计稿" → 追问"有没有参考的产品或截图？希望整体是什么风格？"
- 用户说"用 Ant Design" → 追问"用哪些核心组件？表格、表单、弹窗？有自定义主题色吗？"
- 用户没有明确偏好 → 追问"对色调、间距、字体有要求吗？"

**仅 0-1 新项目且用户未指定 UI 风格时**，追加风格选择提问（使用 AskUserQuestion 提供 4 个选项）：
- 极简轻奢 — 浅色基调、深靛蓝主色、香槟金辅色、大留白、无边框卡片
- 科技未来 — 深色主题、电光紫/赛博蓝、微发光效果、玻璃态面板
- 奢华经典 — 象牙白底、深棕/暗金、衬线标题字体、经典边框
- 自定义 — 用户自行描述风格偏好

### 5. 架构模式（条件触发）

仅已有项目且当前使用平铺目录结构时触发。

入口问题：当前项目使用平铺目录结构（pages/components/api），新功能是否采用 FSD（Feature-Sliced Design）架构？

### 条件触发规则

- **非 0-1 项目**跳过 UI 风格选择
- **已使用 FSD 架构的项目**跳过架构模式选择

### 完成检查表

| # | 维度 | 深挖状态 |
|---|------|---------|
| 1 | 页面需求 | 充分/不足 |
| 2 | 交互需求 | 充分/不足 |
| 3 | 技术栈 | 充分/不足 |
| 4 | UI 约束与风格 | 充分/不足 |
| 5 | 架构模式（条件触发） | 充分/不足/N/A |

**所有适用维度标记为「充分」后才能进入需求确认。** 「不足」的维度必须回去继续追问。

## 需求确认

澄清完成后，向用户展示：

1. **所属上下文** — 说明当前前端开发所属的聚合上下文（即 `<idea-name>` 对应的上下文名称），帮助用户理解前端与后端的对应关系
2. **前端需求总结** — 用自己的话复述前端需求要点
3. **页面列表** — 列出所有需要开发的页面及其功能
4. **API 映射** — 每个页面消费哪些后端 API 端点

使用 AskUserQuestion 询问用户是否确认，或需要调整。

<HARD-GATE>
用户确认前端需求后才能结束澄清流程。
</HARD-GATE>

## 产出模板

将前端需求写入 `.thoughtworks/<idea-name>/frontend-requirement.md`，包含以下章节（按需）：

- **需求要点** — 核心功能描述
- **页面列表** — 页面名称、路由、功能描述
- **API 映射** — 页面与 API 端点的对应关系
- **技术栈** — 确认的前端技术栈
- **UI 约束** — UI 组件库、设计稿等约束信息
- **## UI 风格**（仅 0-1 项目选择了风格时写入） — 记录风格标识（如 `minimalist-luxury`、`tech-futuristic`、`classic-elegant` 或用户自定义描述）
- **## 架构模式**（仅非 0-1 项目时写入） — 记录 `fsd` 或 `flat`

## 合理化预防

| 你可能会想 | 现实 |
|-----------|------|
| "OHS 契约已经说明了一切" | OHS 契约定义了 API，但页面布局、交互流程、UI 风格需要用户确认 |
| "技术栈不用问，用 React 就行" | 如果项目已有前端代码，必须沿用已有技术栈 |
| "直接按 API 端点一对一建页面" | 页面设计应该从用户视角出发，一个页面可能消费多个 API |
| "UI 风格不重要，后面再定" | 0-1 项目的视觉调性会影响 thinker 的组件设计和样式规划，必须在澄清阶段确定 |
| "FSD 太复杂，直接用平铺结构" | FSD 是推荐架构，但对已有项目需尊重用户选择 |
