# 后端需求澄清场景指令

## 项目上下文扫描

在向用户提任何问题之前，先扫描项目现状，目标是了解项目的技术栈、已有代码结构和最近的开发动态，避免问出已有答案的问题：

1. **扫描项目根目录结构** — 使用 Glob 工具查看项目顶层目录和关键配置文件（`pom.xml`、`build.gradle`、`application.yml`、`pyproject.toml`、`requirements.txt`、`go.mod`、`go.sum` 等），目标是确定技术栈和模块划分
2. **读取关键文档** — 使用 Read 工具读取项目的 `README.md`、`CLAUDE.md`（如存在），目标是获取项目背景、架构约定和开发规范
3. **查看最近提交** — 使用 Bash 执行 `git log --oneline -10`，目标是了解最近的开发方向和活跃模块
4. **扫描已有领域模型** — 使用 Glob 查找 `**/domain/**/*.java`、`**/domain/**/*.py`、`**/domain/**/*.go`、`**/entity/**/*.java`，目标是了解已有的领域概念，避免重复设计
5. **检查已有 idea** — 检查 `.thoughtworks/` 目录下是否有相关的已有需求或设计，目标是避免重复工作
6. **检测后端语言** — 根据项目文件特征检测后端语言：
   - `pom.xml` / `build.gradle` / `*.java` → Java
   - `pyproject.toml` / `requirements.txt` / `*.py` → Python
   - `go.mod` / `*.go` → Go
   - 如果检测到多种语言或无法确定，记录下来在澄清阶段询问用户

将扫描结果整理为内部参考（不输出给用户），用于指导后续提问。

## 澄清维度清单

以下维度是**引导锚点**，每个维度都必须深挖到足以指导 DDD 设计的精度，而不是得到一个笼统回答就标记通过。

### 1. 业务目标

入口问题：这个功能要解决什么业务问题？期望的最终效果是什么？

深挖方向：
- 用户说"提高效率" → 追问"具体是哪个环节的效率？当前的痛点是什么？量化的目标是什么？"
- 用户给出功能描述但没说业务动机 → 追问"为什么现在需要这个功能？不做会怎样？"
- 涉及多个目标时 → 追问优先级"如果这两个目标冲突，优先保证哪个？"

### 2. 核心业务规则

入口问题：关键的业务规则是什么？

深挖方向：
- 用户说"价格计算" → 追问"有几种定价策略？折扣如何叠加？最终价格有下限吗？"
- 用户说"状态流转" → 追问"画出完整的状态转换，每个转换的触发条件和前置检查是什么？"
- 用户说"权限控制" → 追问"有几种角色？每个角色的操作权限矩阵是什么？"
- 对每条规则构造边界场景验证，例如"如果同时满足两个互斥条件会怎样？"

### 3. 技术约束

入口问题：有哪些技术约束？

深挖方向：
- 数据库：用什么数据库？有已有的数据表需要兼容吗？
- 外部系统：需要集成哪些外部服务？调用方式（同步/异步）？
- 性能：有并发量要求吗？响应时间要求？
- 用户说"没什么特殊要求" → 主动追问"预计数据量级？是否需要考虑并发？有没有需要对接的第三方系统？"

### 4. 边界与范围

入口问题：哪些是本次范围内的，哪些明确不做？

深挖方向：
- 用户描述了功能但没说边界 → 追问"这些相关功能哪些是本次做的？X 功能是这次做还是后续迭代？"
- 与已有功能有交叉 → 追问"已有 X 功能会受影响吗？需要改造还是只新增？"
- 构造场景探测边界："如果用户在 A 流程中途想做 B，是否需要支持？"

### 5. 成功标准

入口问题：怎样才算做完了？有哪些验收条件？

深挖方向：
- 用户说"功能能用就行" → 追问"具体哪些操作能走通算验收通过？能列出 3-5 个核心场景吗？"
- 没有提到非功能性标准 → 追问"有性能指标要求吗？数据准确性要求？"

### 6. 异常场景

入口问题：异常情况如何处理？

深挖方向：
- 用户没考虑过异常 → 主动构造场景："如果支付成功但库存扣减失败怎么办？需要回滚吗？"
- 用户说"报错就行" → 追问"报错后用户可以重试吗？重试是幂等的吗？需要补偿机制吗？"
- 涉及外部系统 → 追问"外部服务超时或不可用时的降级策略是什么？"

### 7. 后端语言与技术栈

入口问题：确认后端开发语言和核心技术栈。

如果扫描已检测到语言，向用户确认（如"项目检测为 Java Spring Boot，新功能是否继续使用？"）；如果未检测到或检测到多种语言，让用户在 Java（Spring Boot）、Python（FastAPI）、Go（Gin）中选择。

### 完成检查表

| # | 维度 | 深挖状态 |
|---|------|---------|
| 1 | 业务目标 | 充分/不足 |
| 2 | 核心业务规则 | 充分/不足 |
| 3 | 技术约束 | 充分/不足 |
| 4 | 边界与范围 | 充分/不足 |
| 5 | 成功标准 | 充分/不足 |
| 6 | 异常场景 | 充分/不足 |
| 7 | 后端语言与技术栈 | 充分/不足 |

**所有维度标记为「充分」后才能进入领域建模分析。** 「不足」的维度必须回去继续追问。

## 领域建模分析

需求澄清完成后，执行 DDD 战略分析，识别需求中涉及的领域建模单元及其依赖关系。

### DDD 战略分析

基于已澄清的需求和项目上下文，执行以下分析：

1. **识别领域建模单元** — 分析需求涉及的业务概念、业务规则和能力边界，判断适合的建模方式：
   - **聚合**：存在清晰事务边界、一致性要求和生命周期管理时采用
   - **领域服务**：业务规则、决策、计算、校验或策略不适合放入实体/值对象时采用
   - **混合建模**：聚合与领域服务共同承担领域职责时采用

2. **参考已有代码** — 检查扫描到的已有领域模型：
   - 如果新需求涉及已有聚合的扩展，标注为"扩展已有聚合"
   - 如果新需求涉及已有领域服务的扩展，标注为"扩展已有领域服务"
   - 如果新需求与已有领域模型有交叉，明确边界

3. **分析建模单元间依赖关系**：
   - 识别哪些领域建模单元依赖其他单元的数据或能力
   - 标记依赖方向
   - 确定建议的实现顺序（被依赖的单元先实现）

### 特殊情况处理

| 情况 | 处理方式 |
|------|---------|
| 单一建模单元 | 仍执行分析，产出单项方案 |
| 扩展已有聚合或领域服务 | 标注为扩展，说明新增内容 |
| 边界模糊 | 向用户提问确认（"A 和 B 应该收敛为一个聚合，还是拆成两个领域服务？"） |

### 展示领域建模方案

向用户展示领域建模分析结果：

1. **领域建模单元列表**（Markdown 表格）：

| # | 单元名称 | 类型 | 核心职责 | 依赖单元 |
|---|---------|------|---------|---------|
| 1 | Product | 聚合 | 商品基础信息与状态管理 | 无 |
| 2 | InventoryReservation | 领域服务 | 库存预占与释放规则 | Product |
| 3 | Order | 混合 | 订单生命周期与下单规则 | Product, InventoryReservation |

2. **建模单元依赖关系**（文字描述）
3. **建议实现顺序**
4. **各单元职责概要** — 每个建模单元一段简短描述

使用 AskUserQuestion 确认领域建模方案，提供选项：
- 确认方案
- 调整建模（合并/拆分/重命名/调整依赖/调整类型）

## 领域建模方案确认

<HARD-GATE>
用户确认领域建模方案后才能继续。
支持以下调整操作：
- **合并**：将两个建模单元合并为一个
- **拆分**：将一个建模单元拆为多个
- **重命名**：修改建模单元名称
- **调整依赖**：修改建模单元之间的依赖关系
- **调整类型**：在聚合 / 领域服务 / 混合之间切换

用户调整后，重新展示更新的方案，再次确认。
禁止以"领域建模方案已经很合理"为由跳过确认。
</HARD-GATE>

## 产出模板

将需求写入 `.thoughtworks/<idea-name>/requirement.md`，包含以下结构：

```markdown
# <功能名称> 需求文档

## 需求描述

<需求正文>

## 技术选型

- **后端语言**：java / python / go
- **核心框架**：Spring Boot / FastAPI + SQLAlchemy / Gin + GORM

## 领域建模分析

### 识别的领域建模单元

1. **Product**（聚合）：商品管理，包含商品的创建、修改、上下架等核心业务
2. **InventoryReservation**（领域服务）：库存预占、释放与校验规则

### 建模单元依赖关系

- Product：无依赖
- InventoryReservation → Product

### 建议实现顺序

Product → InventoryReservation → Order

## 业务规则

- <规则 1>
- <规则 2>

## 成功标准

- <标准 1>
- <标准 2>
```

## 合理化预防

| 你可能会想 | 现实 |
|-----------|------|
| "不需要做领域建模分析，直接开始设计" | 领域建模分析用于识别边界、职责和依赖，确保 domain 层围绕正确的领域单元设计 |
| "不需要拆分为多目录" | 领域建模分析只识别建模边界和依赖，不创建独立工作流 |
| "已有领域模型不用分析" | 新需求可能需要扩展已有聚合、领域服务或与其交互，必须分析已有领域模型的边界 |
