# Domain 层编码指令

## 编码要求

- **充血模型** — 实体必须包含业务方法，禁止贫血模型
- **信息专家模式** — 谁拥有数据，谁负责逻辑。逻辑跟着数据走
- **受保护的构造 + 工厂方法** — 按 spec 规范中的语言惯用方式
- **字段保护不变性** — 通过业务方法修改状态
- **值对象不可变** — 创建后不可变，业务方法返回新对象
- **仓储接口使用集合语义** — save/remove，不是 insert/delete
- **每个 Repository 方法必须有文档注释**
- **禁止依赖其他层** — 不能导入 infr/ohs/application 模块
- **禁止框架注解/依赖** — Domain 层不依赖任何框架
- **禁止技术细节** — 不能有数据库访问、HTTP 调用、缓存操作、日志记录

## 项目结构探索

在写代码之前，先用 Glob 搜索（根据 CONTEXT 中的 backend_language 选择扩展名）：
- `**/domain/**/*.{ext}` — 找到 domain 包路径（Java: `.java`，Python: `.py`，Go: `.go`）
- `**/pom.xml` 或 `**/build.gradle` 或 `**/pyproject.toml` 或 `**/go.mod` — 确认项目结构

如果找不到已有代码，用 AskUserQuestion 询问用户项目的基础包路径。

## 完成标准

- 所有设计文档中指定的类都已创建
- 代码可编译（import 正确、语法正确）
- 符合 backend-spec domain 规范中的所有约束

## 合理化预防

| 你可能会想 | 现实 |
|-----------|------|
| "这个类太简单，不需要文档注释" | Repository 接口的每个方法都必须有文档注释，这是铁律 |
| "把几个小类放一个文件里更方便" | 按 spec 规范中的语言文件组织规则（Java 每类一个文件，Python/Go 按模块组织） |
| "先用 public 构造函数，后面再改" | 必须使用受保护的构造 + 工厂方法（按 spec 规范），没有"后面" |
| "setter 更方便" | 禁止 setter，通过业务方法修改状态 |
| "加个依赖注入框架注解方便注入" | Domain 层禁止框架注解/依赖 |
