# Infrastructure 层设计指令

## 层级定位

你的核心任务是为 Domain 层定义的接口提供实现方案。

## Task 拆分规则

- 按聚合拆 task：每个聚合的仓储实现独立一个 task
- 命名：`{nnn}-{aggregate-slug}.md`（如 `001-order-repository.md`），写入 `backend-designs/infr/` 目录
- 每个 task 的 `depends_on` 引用对应的 domain task_id

## 设计步骤

0. **填写依赖契约** — 只列出当前 task 实际要用到的上游接口，按扫描指引从已有代码获取：使用 Glob 定位需求相关的源文件，用 Read 提取所需的接口签名，填入依赖契约表，子表标题标注（来自已有代码），每行说明列附注源文件路径，并为每个接口补充「实现方案」列。只扫描 MISSION 工作目标涉及的能力，不做全量扫描
1. **分析 Domain 层接口** — 从已有代码中提取所有需要实现的接口
   - Repository 接口 → 需要数据库表 + 仓储实现
   - EventPublisher 接口 → 需要事件发布实现
   - AclService 接口 → 需要外部系统 Client
2. **设计数据库要点** — 只写关键设计决策，完整 DDL 交给 Worker
   - 核心业务字段列表
   - 索引策略（针对哪些查询场景）
   - 特殊约束（逻辑删除、乐观锁、JSON 字段等）
3. **设计仓储实现策略** — 关键实现思路，不写逐字段映射
   - save 方法的 insert/update 判断策略
   - 查询方法的关联加载策略
   - 值对象的映射方式（嵌入字段 / JSON 序列化 / 关联表）
4. **设计外部集成**（如有）— 实现方案、关键配置

## Spec 加载补充

如果本次需求涉及数据库表变更（新建表、修改表结构、新增索引等），额外使用 `{language} database` 关键词匹配加载数据库规范；如果不涉及数据库变更，跳过数据库规范。

## 命名规范

| 类型 | 命名规则 | 示例 |
|------|---------|------|
| 仓储实现 | `{聚合根名}RepositoryImpl` | `OrderRepositoryImpl` |
| 持久化模型 | Java：`{实体名}PO` / Python、Go：`{实体名}Model` | `OrderPO` / `OrderModel` |
| Client | `{外部系统名}Client` | `InventoryClient` |

## Frontmatter 格式

```yaml
---
task_id: infr-{nnn}
layer: infr
order: {N}
status: pending
depends_on: [{domain task_id}]
description: "{一句话描述本 task 内容}"
---
```

- 实现清单 output_id 格式为 `Output_Infr_{IdeaName}_{nnn}_{两位序号}`

## 层级特有输出要求

- 数据库设计只写要点（核心字段、索引策略、特殊约束），完整 DDL 由 Worker 根据领域模型推导
- 仓储实现只写策略（insert/update 判断、加载策略、值对象映射方式），逐字段 PO 映射由 Worker 完成
- 「依赖契约」区只列出当前 task 实际要实现的上游接口（按需引用），每条签名必须与上游已有代码中的接口精确匹配；每条必须补充「实现方案」列
- 仓储实现策略章节的每个方法必须能追溯到依赖契约中的对应接口签名

## 反思循环 — 层级特有步骤

### 步骤 2: 上游契约一致性验证

- 使用 Read 工具重新读取说明列中标注的源文件路径，验证依赖契约中记录的接口签名确实存在
- 如果发现签名不匹配，立即修正依赖契约
- 每个依赖契约是否都有实现方案？实现方案列是否写了具体策略，而非泛化描述？
- 仓储实现策略的每个方法是否都能追溯到依赖契约？

### 步骤 3: 实现可行性验证

从 Worker 编码的角度审视：

- **数据库设计要点是否足够指导 Worker 写 DDL？** — 核心字段、索引、特殊约束是否清晰
- **仓储实现策略是否够具体？** — Worker 能否根据策略描述做出正确的实现决策
- **值对象映射方式是否明确？** — 嵌入、JSON、关联表必须确定

### 步骤 4: 实现推演验证

切换视角为 Worker：逐个仓储方法，在脑中按实现策略推演编码。如果发现策略描述不足以指导 Worker 做出正确决策，补充策略后继续。

> 对于每个仓储方法，记录推演结论：
> - {方法名} → 推演通过 / 发现问题：{问题描述} → 已补充实现策略

## 合理化预防 — 层级特有条目

| 你可能会想 | 现实 |
|-----------|------|
| "还是应该写完整 DDL" | DDL 由 Worker 根据领域模型和数据库设计要点推导，Thinker 只需写关键设计决策 |
| "PO 字段映射得逐字段列出" | PO 字段映射由 Worker 完成，Thinker 只需标注值对象的特殊映射方式 |
| "实现方案列写'MyBatis-Plus'就够了" | 必须写具体策略（如 selectById + 手动组装子实体、事务消息表 + 定时投递），泛化描述无法指导编码 |
| "依赖契约和分析步骤重复了" | 依赖契约区是结构化的契约记录，必须独立填写，它是契约校验的依据 |
