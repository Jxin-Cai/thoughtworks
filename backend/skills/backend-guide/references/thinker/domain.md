# Domain 层设计指令

## 层级定位

你的产出只交给本层的 Worker 实现。其他层通过扫描已实现代码获取跨层依赖。

## 设计步骤

1. **分析需求** — 识别核心业务概念和业务规则
2. **选择领域建模单元** — 判断哪些业务规则应进入实体/值对象，哪些需要清晰聚合边界，哪些应由领域服务承载
   - 有清晰一致性边界、生命周期和状态封装要求时优先设计聚合
   - 不适合进入实体/值对象、但明确属于领域层的规则、决策、计算、校验与策略，应设计为领域服务
3. **设计充血模型** — 每个实体必须包含业务方法，禁止贫血模型
   - 受保护的构造 + 工厂方法（按 spec 规范中的语言惯用方式）
   - 字段保护不变性
   - 通过业务方法修改状态
   - 优先将业务规则沉到实体、值对象或领域服务
4. **定义仓储接口** — 有明确持久化边界时再定义仓储，使用集合语义（save/remove）
   - 每个方法必须有文档注释：做什么、关键约束、异常场景
   - 入参只能是领域模型或 ID，返回值必须是领域模型或其可选包装
5. **识别领域事件** — 确定触发时机、携带数据、消费方预期
6. **识别防腐层接口** — 如需调用外部领域，定义 ACL 接口
7. **识别领域服务** — 跨聚合逻辑，以及不适合放入实体/值对象但明确属于领域层的规则、决策、计算、校验与策略，都由领域服务承载
8. **填写导出契约** — 汇总聚合根/实体 API、值对象定义、领域服务 API、仓储/事件发布/防腐层接口签名，供 Worker 实现和契约校验引用

## 导出契约设计要点

确保：
- **Repository 接口**方法签名清晰 → Infrastructure 层需要实现
- **领域对象/领域服务业务方法**签名明确 → Application 层需要编排调用
- **领域事件**定义完整 → Infrastructure 层需要实现发布机制

## 命名规范

| 类型 | 命名规则 |
|------|---------|
| 聚合根/实体 | `{业务概念名}` |
| 值对象 | `{业务概念名}` |
| 领域服务 | `{业务名}{动作}Service` |
| 仓储接口 | `{聚合根名}Repository` |
| 领域事件 | `{聚合根名}{动作过去式}Event` |
| 事件发布接口 | `{聚合根名}EventPublisher` |
| 防腐层接口 | `{外部领域名}AclService` |

## Task 拆分规则

- 按领域建模单元拆 task：一个聚合、一个核心领域服务，或一组强相关的小型建模单元可组成一个 task
- 命名：`{nnn}-{domain-topic-slug}.md`（如 `001-order-domain.md`、`002-pricing-policy.md`），写入 `backend-designs/domain/` 目录
- task 应围绕清晰的领域主题组织，单个 task ≤800 行
- domain task 通常无依赖（`depends_on: []`）

## Frontmatter 格式

```yaml
---
task_id: domain-{nnn}
layer: domain
order: {N}
status: pending
depends_on: []
description: "{一句话描述本 task 内容}"
---
```

- 实现清单 output_id 格式为 `Output_Domain_{IdeaName}_{nnn}_{两位序号}`（IdeaName 取 idea-name 的 PascalCase），序号从 01 开始

## 层级特有输出要求

- 每个 Repository 方法必须有完整的文档注释
- 所有字段必须标注类型和约束
- 「导出契约」区必须完整填写，它是 Worker 实现和契约校验的依据
- 接口签名表的「行为描述」列必须包含具体策略（如 save 的 insert/update 策略、查询的加载策略、事件的投递语义），不能只写"保存实体"这类泛化描述
- 导出契约表中的每个签名必须能在正文中找到对应定义，禁止出现正文没有但契约表有的签名

## 反思循环 — 层级特有步骤

### 步骤 2: 导出契约完备性验证

审视导出契约的完整性和可实现性：

- **导出契约完备吗？** — 正文中出现的每个接口/类型，在导出契约表中都有对应条目吗？
- **签名足够具体吗？** — Worker 只看导出契约，能否无歧义地实现？
- **行为描述够详细吗？** — 策略性描述（insert/update 判断、加载策略等）是否足够指导编码？

### 步骤 3: 规范符合性验证

对照 backend-spec 规范和本设计步骤检查：

- 命名是否符合命名规范表？
- 充血模型是否完整（受保护的构造、工厂方法、字段不变性保护）？
- Repository 方法是否都有文档注释？
- 是否存在贫血模型、技术侵入、命名不规范等问题？

### 步骤 4: 实现推演验证

切换视角为 Worker：逐个方法，在脑中按方案的描述写出完整实现代码。如果写到某一行时发现方案的描述在特定数据状态下会导致运行时异常或产生错误结果，说明方案本身有缺陷 — 补充处理策略后再继续。

> 对于每个方法/接口签名，记录推演结论：
> - {方法名} → 推演通过 / 发现问题：{问题描述} → 已补充处理策略

## 合理化预防 — 层级特有条目

| 你可能会想 | 现实 |
|-----------|------|
| "这个章节不适用，跳过" | 模板中没有的章节才能跳过，有的章节必须填写或显式标注"不需要" |
| "方法签名后面再细化" | 现在就必须写到参数类型和返回类型，Worker 依赖你的签名直接编码 |
| "Repository 的文档注释太啰嗦" | 每个方法必须有文档注释，Infrastructure 层靠它理解实现预期 |
| "值对象太简单不需要业务方法" | 如果值对象有验证规则或转换逻辑，必须设计业务方法 |
| "字段约束显而易见不用写" | 所有字段必须标注类型和约束，不能假设 Worker 能自行推断 |
| "行为描述写'保存实体'就够了" | 必须写具体策略（insert/update 判断、加载策略、投递语义），Worker 靠它决定实现方案 |
