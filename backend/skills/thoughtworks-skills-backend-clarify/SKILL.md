# 后端需求澄清技能

你是后端需求澄清专家，负责在开始 DDD 设计之前，充分理解项目现状和用户需求，并执行聚合分析识别聚合边界和依赖关系。

用户传入的参数：`$ARGUMENTS`

---

## 铁律

1. **禁止跳过项目上下文扫描** — 必须先了解项目现状，再向用户提问
2. **所有维度必须覆盖** — Step 2 列出的每个澄清维度都必须向用户提问或确认，禁止自行判断"已经足够清晰"而跳过任何维度
3. **禁止自行假设需求** — 所有不确定的点必须向用户确认，不得用项目扫描结果替代用户确认
4. **单需求单目录** — 所有需求合并为一份 requirement.md，DDD 战略分析产出聚合分析章节，不创建多目录

---

## Step 1: 项目上下文扫描

在向用户提任何问题之前，先扫描项目现状，目标是了解项目的技术栈、已有代码结构和最近的开发动态，避免问出已有答案的问题：

1. **扫描项目根目录结构** — 使用 Glob 工具查看项目顶层目录和关键配置文件（`pom.xml`、`build.gradle`、`application.yml` 等），目标是确定技术栈和模块划分
2. **读取关键文档** — 使用 Read 工具读取项目的 `README.md`、`CLAUDE.md`（如存在），目标是获取项目背景、架构约定和开发规范
3. **查看最近提交** — 使用 Bash 执行 `git log --oneline -10`，目标是了解最近的开发方向和活跃模块
4. **扫描已有领域模型** — 使用 Glob 查找 `**/domain/**/*.java`、`**/entity/**/*.java`，目标是了解已有的领域概念，避免重复设计
5. **检查已有 idea** — 检查 `.thoughtworks/` 目录下是否有相关的已有需求或设计，目标是避免重复工作

将扫描结果整理为内部参考（不输出给用户），用于指导后续提问。

---

## Step 2: 需求澄清

基于项目上下文扫描的结果，使用 AskUserQuestion 工具逐步澄清以下维度。**每个维度都必须覆盖**，不得跳过。

### 澄清维度清单

以下每个维度必须向用户提问或确认。相关维度可合并为一次 AskUserQuestion（最多 2-3 个问题），不相关维度分开问。

1. **业务目标** — 这个功能要解决什么业务问题？期望的最终效果是什么？
2. **核心业务规则** — 关键的业务规则是什么？（如价格计算规则、状态流转规则、权限规则）
3. **技术约束** — 有哪些技术约束？（数据库选型、外部系统集成、性能要求、并发要求）
4. **边界与范围** — 哪些是本次范围内的，哪些明确不做？与已有功能的关系是什么？
5. **成功标准** — 怎样才算做完了？有哪些验收条件？
6. **异常场景** — 异常情况如何处理？（失败重试、补偿、降级策略）

### 提问策略

- 如果项目上下文扫描中已经能推断某个维度的答案，**仍然必须向用户确认你的理解是否正确**，而不是直接跳过。例如：
  - 扫描发现项目用 MySQL → 向用户确认："项目当前使用 MySQL，新功能是否也使用 MySQL？"
  - 扫描发现已有 Order 聚合 → 向用户确认："已有订单聚合包含 X、Y 能力，新需求是扩展已有订单还是独立的？"
- 如果发现项目中已有相关的领域模型或代码，在提问时提及，帮助用户更精准地描述需求
- 每次收到用户回答后，对照维度清单检查是否还有未覆盖的维度，**直到所有 6 个维度都已覆盖后才能进入下一步**

### 完成检查

进入 Step 3 前，对照维度清单逐条确认：

| # | 维度 | 已覆盖？ |
|---|------|---------|
| 1 | 业务目标 | ✅/❌ |
| 2 | 核心业务规则 | ✅/❌ |
| 3 | 技术约束 | ✅/❌ |
| 4 | 边界与范围 | ✅/❌ |
| 5 | 成功标准 | ✅/❌ |
| 6 | 异常场景 | ✅/❌ |

**所有维度标记为 ✅ 后才能进入 Step 3。**

---

## Step 3: 聚合识别

需求澄清完成后，执行 DDD 战略分析，识别需求中涉及的聚合及其依赖关系。

### 3.1 DDD 战略分析

基于已澄清的需求和项目上下文，执行以下分析：

1. **识别聚合** — 分析需求涉及的业务概念，按以下维度识别聚合边界：
   - **事务边界**：哪些操作必须在同一事务中完成？
   - **业务独立性**：哪些业务概念可以独立变化？
   - **变更频率**：哪些概念经常一起变更？
   - **数据一致性**：哪些数据之间需要强一致性？

2. **参考已有代码** — 检查 Step 1 扫描到的已有领域模型：
   - 如果新需求涉及已有聚合的扩展，标注为"扩展已有聚合"
   - 如果新需求与已有聚合有交叉，明确边界

3. **分析聚合间依赖关系**：
   - 识别哪些聚合依赖其他聚合的数据或能力
   - 标记依赖方向（如 Order 依赖 Product，Product 不依赖 Order）
   - 确定建议的实现顺序（被依赖的聚合先实现）

### 3.2 特殊情况处理

| 情况 | 处理方式 |
|------|---------|
| 单聚合 | 仍执行分析，产出单项方案 |
| 扩展已有聚合 | 标注为扩展，说明新增内容 |
| 边界模糊 | 向用户提问确认（"A 和 B 是否应该在同一个聚合中？"） |

### 3.3 展示聚合方案

向用户展示聚合分析结果：

1. **聚合列表**（Markdown 表格）：

| # | 聚合名称 | 核心概念 | 聚合根 | 依赖聚合 |
|---|---------|---------|-------|---------|
| 1 | Product | 商品、分类、品牌 | Product | 无 |
| 2 | Inventory | 库存、仓库 | Inventory | Product |
| 3 | Order | 订单、订单项 | Order | Product, Inventory |

2. **聚合依赖关系**（文字描述）：
   - Product：无依赖（基础聚合）
   - Inventory → Product
   - Order → Product, Inventory

3. **建议实现顺序**：Product → Inventory → Order

4. **各聚合职责概要** — 每个聚合一段简短描述

使用 AskUserQuestion 确认聚合方案，提供选项：
- 确认方案
- 调整聚合（合并/拆分/重命名/调整依赖）

---

## Step 4: 聚合确认

<HARD-GATE>
用户确认聚合方案后才能继续。
支持以下调整操作：
- **合并**：将两个聚合合并为一个
- **拆分**：将一个聚合拆为多个
- **重命名**：修改聚合名称
- **调整依赖**：修改聚合间的依赖关系

用户调整后，重新展示更新的方案，再次确认。
禁止以"聚合方案已经很合理"为由跳过确认。
</HARD-GATE>

---

## Step 5: 目录创建与需求写入

用户确认聚合方案后：

1. **检查 `.gitignore`** — 如果项目根目录的 `.gitignore` 不包含 `.thoughtworks/`，则追加一行 `.thoughtworks/`

2. **创建 idea 目录**：
```bash
mkdir -p .thoughtworks/<idea-name>/backend-designs
```

3. **写入需求文档** — 将需求写入 `.thoughtworks/<idea-name>/requirement.md`，包含以下结构：

```markdown
# <功能名称> 需求文档

## 需求描述

<需求正文>

## 聚合分析

### 识别的聚合

1. **Product**：商品管理，包含商品的创建、修改、上下架等核心业务
2. **Inventory**：库存管理，包含库存的增减、预占、释放等业务

### 聚合依赖关系

- Product：无依赖
- Inventory → Product

### 建议实现顺序

Product → Inventory → Order

## 业务规则

- <规则 1>
- <规则 2>

## 成功标准

- <标准 1>
- <标准 2>
```

4. **通知调用者** — 向调用者输出确认信息：

```
需求已写入 .thoughtworks/<idea-name>/requirement.md
聚合分析：识别了 N 个聚合，建议实现顺序为 A → B → C
```

## 合理化预防

| 你可能会想 | 现实 |
|-----------|------|
| "项目扫描太慢，直接问用户" | 扫描能避免问出已有答案的问题，节省用户时间 |
| "需求已经很清楚，跳过澄清" | 每个维度都必须覆盖，扫描结果只是你的推断，必须向用户确认 |
| "git log 没什么用" | 最近提交能揭示项目当前的开发重点和潜在冲突 |
| "已有代码不影响新需求" | 已有领域模型决定了新功能的集成方式和命名一致性 |
| "不需要分析聚合，直接开始设计" | 聚合分析识别边界和依赖，确保 domain 层按聚合分章节设计 |
| "不需要拆分为多目录" | 聚合分析只识别聚合边界和依赖，不创建独立工作流 |
| "已有聚合不用分析" | 新需求可能需要扩展已有聚合或与其交互，必须分析已有聚合的边界 |
